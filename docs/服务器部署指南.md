## 2. 本地准备

### 2.1 服务器环境变量

复制模板并填写真实值：

```bash
cp deploy/.env.server.example deploy/.env.server
```

这个文件用于远端容器运行，包含：

- PostgreSQL 连接信息
- Redis 和 Qdrant 地址
- Claude / 智谱配置
- OSS 配置
- `SECRET_KEY`
- `ADMIN_PHONES`

`deploy/.env.server` 已被 `.gitignore` 忽略，不应提交。

### 2.2 部署连接参数

复制部署参数模板：

```bash
cp deploy/.deploy.env.example deploy/.deploy.env
```

当前推荐配置：

```bash
SERVER_HOST=47.123.7.235
SERVER_USER=root
SERVER_PORT=22
SERVER_PASSWORD=

REMOTE_DIR=/srv/toolStore
HEALTHCHECK_URL=http://47.123.7.235/health
```

说明：

- 如果使用密码登录，把 `SERVER_PASSWORD` 只填写在本地 [deploy/.deploy.env](/Users/weize/Desktop/work/toolStore/deploy/.deploy.env)
- 更推荐后续切换为 SSH key，这样可以一直保持 `SERVER_PASSWORD` 为空

## 3. 一键部署脚本

部署脚本位置：

- [deploy/deploy_server.sh](/Users/weize/Desktop/work/toolStore/deploy/deploy_server.sh)

执行前先给权限：

```bash
chmod +x deploy/deploy_server.sh
```

### 3.1 默认更新部署

```bash
./deploy/deploy_server.sh
```

脚本会自动完成：

1. 本地打包项目
2. 上传发布包到服务器
3. 保留远端已有 `deploy/.env.server`
4. 在远端重建并启动 Docker Compose
5. 访问 `/health` 做部署后验证

### 3.2 同步远端环境文件

如果要把本地最新 `deploy/.env.server` 一并上传到服务器：

```bash
./deploy/deploy_server.sh --sync-env
```

注意：

- 这个操作会用本地环境文件覆盖远端运行环境
- 只有在你明确确认本地 `deploy/.env.server` 是最新正确版本时才这样做

### 3.3 临时传入密码

如果你不想把密码写进 `deploy/.deploy.env`：

```bash
./deploy/deploy_server.sh --password '你的服务器密码'
```

### 3.4 常用参数

```bash
./deploy/deploy_server.sh --host 47.123.7.235
./deploy/deploy_server.sh --sync-env
./deploy/deploy_server.sh --skip-healthcheck
```

## 4. 远端手工命令

如果自动脚本不可用，也可以在服务器上手工操作：

```bash
cd /srv/toolStore/deploy
docker compose --env-file .env.server -f docker-compose.server.yml up -d --build
docker compose --env-file .env.server -f docker-compose.server.yml ps
```

查看日志：

```bash
docker compose --env-file .env.server -f docker-compose.server.yml logs -f app
```

停止服务：

```bash
docker compose --env-file .env.server -f docker-compose.server.yml down
```

重启应用：

```bash
docker compose --env-file .env.server -f docker-compose.server.yml restart app
```

## 5. 首次部署检查项

首次部署前至少确认下面几项：

- 服务器已安装 Docker 和 Docker Compose
- 80 端口未被业务 Nginx 或其他进程占用
- 服务器能够访问 Python 包镜像源
- `deploy/.env.server` 已填写完整
- OSS / AI / 数据库参数已可用

当前脚本会尝试清理占用 80 端口的遗留 `nginx` 进程，但如果 80 被其他业务占用，脚本会停止，不会强行覆盖。

## 8. 回滚方式

当前自动脚本在远端部署时会把旧目录改名为时间戳备份：

```bash

```

如果新版本有问题，可以在服务器上手工回滚：

```bash

mv /srv/toolStore /srv/toolStore.failed.$(date +%Y%m%d%H%M%S)
mv /srv/toolStore.bak.<timestamp> /srv/toolStore
cd /srv/toolStore/deploy

```

## 10. 常见故障

### 10.1 80 端口被占用

现象：

- `toolstore-app` 无法启动
- Docker 报 `bind: address already in use`

检查：

```bash
ss -ltnp | grep ':80 '
```

处理：

- 如果是遗留 `nginx`，停止它
- 如果是正式业务服务占用，不要强停，先调整端口或接反向代理

### 10.2 应用容器启动后立刻退出

检查：

```bash
docker logs --tail 100 toolstore-app
```

常见原因：

- 环境变量缺失
- 数据库连接错误
- 启动种子数据异常
- OSS 或 AI 配置不合法

### 10.3 `/health` 正常但移动端连不上

优先检查：

- 移动端是否仍携带旧的 `--dart-define=API_BASE_URL`
- 设备网络是否能访问 `47.123.7.235`
- 是否存在跨网段或公司网络限制
